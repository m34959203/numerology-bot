## Принцип разделения

- **bot/** ничего не знает о нумерологии — собирает анкету, проводит оплату, зовёт `core` и форматирует ответ.
- **core/numerology/** — чистые функции (детерминированные, тестируемые без Telegram/БД). Расчётные модули = перенос листов Excel; `tariffs.py` и `daily.py` — продуктовый слой поверх расчёта.
- **core/content/** — данные (тексты-трактовки), а не логика.

## Поток данных

1. `catalog` → пользователь выбирает тариф. Ручные тарифы (детские, совместимость) ведут к мастеру, остальные — к оплате.
2. `payment` → invoice Telegram Stars → `pre_checkout` → `successful_payment` (запись в `payments` с `telegram_payment_charge_id`). В тестовом режиме (`PAYMENT_IMITATION`) платёж имитируется.
3. `survey` (FSM) → собирает ФИО + даты (доп. поля — по требованиям тарифа) → запись в `survey_data`.
4. `tariffs.report_for(person, ref, code)` → `report.build_report()` собирает нужные секции, прикрепляет `_fields` (атомы тарифа) → `results`.
5. `render`/`pdf` → выдача (текст с разбивкой по 4096 / премиальный PDF), доступ через «Мои расчёты».

## Дифференциация тарифов (выбор по полям)

`core/numerology/tariffs.py` задаёт для каждого кода услуги `TariffSpec(fields, forecast_years, manual, needs_parents, needs_name)`. `fields` — набор **атомов** (ЧЖП, число души, финкод, прогноз, кодировки…). `report_for` собирает секции по атомам и кладёт в отчёт `_fields`. Рендереры показывают под-блок только если его атом в `_fields` (`_want`); без `_fields` показывают всё. Ручные тарифы (`manual=True`) не считаются — `catalog` ведёт к контакту мастера.

## Трактовки (лукап по значению)

Каждое описание — VLOOKUP по рассчитанному значению (`core/content/loader.interpret`): меняется значение → подтягивается другое описание. В прогнозе значения меняются по годам/дням, поэтому трактовки пересчитываются на каждый период (`report.build_forecast_section` обогащает каждый год: итог, нулевой период, график кода жизни). Тексты извлекаются из листа «текст» Excel скриптом `scripts/extract_texts.py` (карта диапазонов — [content-extraction-map.md](content-extraction-map.md)).

## Прогноз на день

`core/numerology/daily.py` `daily_forecast(person, target_date)` — интерактивная функция (нужна только дата рождения): ЧПД + трактовка, личный год/месяц, статус биоритма. Флоу — `bot/handlers/daily.py` (меню «Прогноз на день» / `/day`), `reference_date` = выбранная дата.

## Идемпотентность платежей

`telegram_payment_charge_id` уникален → повторный `successful_payment` не создаёт второй выдачи. Возврат — `refund_star_payment` админом, статус платежа `refunded`.

## Перенос Excel

Логика восстанавливается по листам целиком (не по ячейкам). Соответствие лист↔модуль — в [CLAUDE.md](../CLAUDE.md). Excel-функции воспроизводятся хелперами в чистом Python.
